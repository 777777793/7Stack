## Q：如何设计一个秒杀系统，既抗住高并发又绝不超卖？
### 📌 一句话结论
秒杀系统的核心不是“把请求扛住”，而是“把无效请求尽早挡在前面，把有效请求变成有序处理”，通常采用“静态化 + 多级限流 + Redis 预扣库存 + MQ 异步下单 + 最终一致性补偿”的组合拳。

### 🧠 原理讲解
秒杀的本质像一扇只能通过 100 人的门，但外面一瞬间来了 10 万人。真正危险的不是人多，而是所有人一起冲门，导致数据库锁冲突、线程池打满、库存被并发扣错。

所以高级设计的思路不是让 10 万人都进入业务逻辑，而是分层过滤：
- 第一层先挡掉绝大多数无资格流量，比如机器刷子、重复点击、没登录用户。
- 第二层只让少量“可能成功”的请求进入核心链路。
- 第三层把“扣库存”从数据库前移到 Redis，先用内存做高速原子判断。
- 第四层把“创建订单”异步化，避免数据库在瞬时洪峰下被打穿。

一句话，秒杀是典型的“用空间换时间，用异步换吞吐，用前置校验换后端稳定”。

### ✅ 完整解决方案
#### 1. 活动预热与页面静态化
先把商品详情页、库存展示页、活动页静态化，挂 CDN。

这么做的原因很直接：
- 秒杀开始前，绝大部分流量都只是查看页面，不应该打到应用层。
- 静态资源走 CDN，可以把源站压力提前卸掉。
- 活动开始前把秒杀商品、库存阈值、风控规则预热到 Redis，避免临时加载。

#### 2. 资格校验前置，先拦垃圾流量
请求真正进入秒杀逻辑前，先做三类校验：
- 登录态校验，未登录直接拦。
- 路径动态化，秒杀 URL 不是固定路径，而是带签名或一次性 token。
- 风控校验，做人机验证、设备指纹、IP 限速、用户维度限购。

这是第一道闸门。很多系统挂掉，不是因为库存不够，而是因为让无效流量也把线程、连接池、Redis 都占满了。

#### 3. 多级限流，把洪峰切平
秒杀限流至少要三层：
- Nginx/OpenResty 层做 IP 级粗粒度限流，挡恶意洪水。
- 网关层做用户维度、接口维度限流，保护下游。
- 应用层做热点商品维度限流，例如单个 sku 每秒最多放行 5000 个请求。

Trade-off：
- 限流阈值太小，损失真实成交。
- 限流阈值太大，核心服务会被打穿。

所以实际做法通常不是“全量放行再失败”，而是根据压测结果给每一层设置兜底阈值，并保留动态配置能力。

#### 4. Redis 预扣库存，先解决超卖问题
核心库存不能直接在 MySQL 上做 `update stock = stock - 1 where stock > 0` 硬抗，因为热点行会被锁成串行，吞吐极低。

更优解是：
- 秒杀开始前，把可售库存加载到 Redis。
- 每次请求先在 Redis 里用 Lua 原子扣减。
- 扣减成功才有资格发消息进入下单链路。
- 扣减失败直接返回“已售罄”。

这里 Redis 不是最终库存，而是“流量闸门库存”。数据库库存是最终账本，后面通过异步订单和补偿保证一致。

#### 5. 一人一单控制，避免重复下单
秒杀里常见坑是用户疯狂点击，结果成功扣了多次库存。

典型做法：
- Redis 里维护 `userId + skuId` 的抢购标记。
- Lua 脚本里把“判断库存”和“判断是否重复购买”放在一次原子操作里完成。
- 标记成功后再发 MQ。

这一步必须和预扣库存一起原子执行，否则就会出现“库存扣了但用户资格未锁住”的并发漏洞。

#### 6. MQ 异步下单，削峰填谷
Redis 扣库存成功后，不要同步创建订单，而是：
- 只返回“抢购请求已受理”。
- 发送下单消息到 MQ。
- 消费者串行或分片消费，执行创建订单、写 DB、冻结库存、支付流程。

这样做的价值：
- 前台接口 RT 很短，能扛住更大并发。
- 下游数据库不再承受瞬时尖峰，而是被 MQ 平滑掉。
- 失败时便于重试和补偿。

#### 7. 数据库最终落单与补偿回滚
真正的订单创建成功后：
- 写订单表。
- 写库存流水或冻结记录。
- 更新用户购买记录。

如果消费失败，不能简单丢掉，要做补偿：
- 创建订单失败但 Redis 已预扣，要回补 Redis 库存。
- MQ 长时间失败要进死信队列，人工或任务补偿。
- 支付超时未支付，要解锁库存或走超时关闭逻辑。

这里体现的是架构思维：秒杀成功不是一条 SQL，而是一条状态机。

#### 8. 防止库存不一致的关键策略
高级系统通常会把库存分成三层：
- 活动总库存：运营定义上限。
- Redis 可抢库存：入口闸门。
- DB 实际库存 / 冻结库存：最终账本。

再配合两类补偿：
- 正向补偿：Redis 扣了但订单没建成，回补。
- 逆向补偿：订单关闭或支付失败，释放冻结库存。

如果面试官追问“怎么保证绝对不超卖”，你要明确回答：
- 单机层面靠 Lua 原子扣减防并发竞争。
- 分布式层面靠预扣资格控制进入人数。
- 最终账本层面靠数据库事务和库存状态流转。
- 极端异常下允许少卖，不允许超卖。

这是成熟电商最常见的取舍。

#### 9. 青铜到王者的演进路径
- 青铜版：数据库直接扣库存。
- 白银版：Redis 扣库存 + MySQL 下单。
- 黄金版：Redis 预扣 + MQ 异步下单 + 一人一单。
- 王者版：再加动态限流、灰度开关、热点隔离、风控、人机验证、监控补偿、库存对账。

### 💻 核心代码
```java
@RestController
@RequestMapping("/seckill")
public class SeckillController {

    private final StringRedisTemplate redisTemplate;
    private final RocketMQTemplate rocketMQTemplate;

    // Lua 脚本：同时校验库存和一人一单，保证原子性
    private static final DefaultRedisScript<Long> SECKILL_SCRIPT = new DefaultRedisScript<>(
            """
            local stockKey = KEYS[1]
            local userKey = KEYS[2]
            local userId = ARGV[1]
            local orderMsg = ARGV[2]

            local stock = tonumber(redis.call('get', stockKey))
            if not stock or stock <= 0 then
                return 1 -- 库存不足
            end

            if redis.call('sismember', userKey, userId) == 1 then
                return 2 -- 重复购买
            end

            redis.call('decr', stockKey)      -- 原子扣减库存
            redis.call('sadd', userKey, userId) -- 标记用户已抢购
            redis.call('xadd', 'stream.orders', '*', 'msg', orderMsg) -- 可替换为 MQ/outbox
            return 0
            """,
            Long.class
    );

    @PostMapping("/{skuId}")
    public ResponseEntity<String> seckill(@PathVariable Long skuId, @RequestHeader("uid") Long userId) {
        String stockKey = "seckill:stock:" + skuId;
        String userKey = "seckill:buyers:" + skuId;
        String orderMsg = userId + ":" + skuId;

        Long result = redisTemplate.execute(
                SECKILL_SCRIPT,
                List.of(stockKey, userKey),
                String.valueOf(userId),
                orderMsg
        );

        if (result == null || result == 1L) {
            return ResponseEntity.status(429).body("Sold out");
        }
        if (result == 2L) {
            return ResponseEntity.badRequest().body("Duplicate request");
        }

        // 生产上更常见的是 Lua 只做预扣，发送 MQ 放在应用层或 outbox 保证可靠性
        rocketMQTemplate.convertAndSend("seckill-order-topic", new SeckillOrderCommand(userId, skuId));
        return ResponseEntity.ok("Queued");
    }
}
```

### ⚠️ 常见误区
- 误区 1：以为 `update ... where stock > 0` 就能搞定秒杀。它能防超卖，但扛不住热点行锁竞争，吞吐很差。
- 误区 2：Redis 扣库存成功后直接返回“下单成功”。其实只是“抢购资格成功”，订单最终还要看异步落单结果。
- 误区 3：只做库存扣减，不做一人一单原子控制。结果会出现同一用户重复抢到库存。

### 🎤 面试简答版
秒杀系统我一般会拆成四层：入口静态化、资格和风控前置、多级限流、核心链路异步化。库存不会直接打 MySQL，而是先在 Redis 用 Lua 做原子预扣，同时控制一人一单，成功后再发 MQ 异步创建订单。数据库只做最终账本和状态流转，失败靠补偿回滚。这个方案本质上是尽量把无效请求挡在外面，把有效请求变成有序处理，宁可少卖也不能超卖。

## Q：如何设计一套多级限流体系，防止系统被突发流量打垮？
### 📌 一句话结论
限流不是单点技术，而是一整套分层保护机制，通常要做“边缘层粗限流 + 网关层通用限流 + 应用层热点限流 + 下游线程池隔离”，并配合降级和动态开关形成闭环。

### 🧠 原理讲解
限流本质上是在问一个问题：系统现在最多只能吃 1 万个请求，但外面来了 5 万个，你准备让哪 4 万个死在什么地方？

如果你不主动做决策，最后就会变成最糟糕的被动结果：
- 线程池耗尽。
- 数据库连接池打满。
- Redis RT 飙升。
- 整个服务雪崩式超时。

所以限流的目标不是“一个请求都不能丢”，而是“有秩序地丢掉一部分请求，保证核心流量活下来”。

### ✅ 完整解决方案
#### 1. 先做容量建模，再谈限流阈值
限流阈值不是拍脑袋定的，要基于压测反推：
- 单机最大 QPS。
- 集群总 QPS。
- Redis / DB / MQ 等下游瓶颈。
- 核心接口和非核心接口的优先级。

比如网关层能扛 5 万 QPS，不代表订单服务能扛 5 万 QPS，因为真正的瓶颈可能在库存服务和 DB。

#### 2. 第一层：Nginx / OpenResty 粗粒度限流
这一层主要拦两类流量：
- 明显异常流量，比如某个 IP 短时疯狂请求。
- 突发洪峰，把最粗的流量先砍掉一批。

适合用漏桶、令牌桶等算法。

Trade-off：
- 边缘层性能最好，但业务语义最弱。
- 它适合挡“洪水”，不适合做复杂的用户级限购。

#### 3. 第二层：网关层通用限流
网关层适合做：
- 用户维度限流。
- 接口维度限流。
- 租户维度限流。
- 降级兜底，比如直接返回“系统繁忙，请稍后重试”。

这一层的价值在于统一治理。比如 Sentinel、Envoy、Spring Cloud Gateway 都能做比较通用的限流与熔断。

#### 4. 第三层：应用层热点参数限流
真正高价值的限流，往往在应用层。

典型场景：
- 秒杀接口里，某个热点 sku 被瞬间打爆。
- 热门直播间的发言、点赞、红包接口流量远高于普通房间。
- 某个商家或租户突然爆量。

这时候必须按“业务热点对象”限流，而不是只按 URL 限流。

比如：
- `skuId`
- `roomId`
- `tenantId`
- `userId`

这样做才能精准保护热点资源，而不是把整个系统一刀切死。

#### 5. 线程池隔离与舱壁模式
只做请求入口限流还不够，服务内部也要做隔离：
- 核心接口独立线程池。
- 慢依赖和快依赖分开线程池。
- 非核心功能，比如埋点、推荐、画像查询走独立资源池。

原因很简单：如果一个非核心依赖把线程都卡死，核心下单链路也会被拖死。

这其实是“限流 + 隔离 + 降级”三件事联动。

#### 6. 降级策略一定要提前设计
限流之后，不能只返回一个 500，要有明确业务降级：
- 评论服务挂了，先隐藏评论区。
- 推荐服务超时，返回默认推荐。
- 发券接口过载，排队或返回稍后重试。
- 秒杀流量过大，直接展示“已售罄”而不是继续打后端。

真正成熟的系统，降级页面和提示文案都是提前演练过的。

#### 7. 动态配置与熔断联动
高并发场景下，限流阈值必须支持动态调整：
- 热点活动开始前临时提阈值。
- 下游 Redis / DB RT 升高时自动下调阈值。
- 某个依赖异常时快速熔断，避免雪崩。

这就是治理平台的价值。限流不是写死在代码里，而是能实时调参。

#### 8. 常见限流算法如何取舍
##### 固定窗口
实现简单，但窗口切换时会有突刺问题。

##### 滑动窗口
更平滑，适合需要严格控制时间窗流量的场景，但实现和存储开销更高。

##### 漏桶
强调匀速出流，适合下游处理能力固定的场景。

##### 令牌桶
允许一定突发，更贴近互联网接口限流的真实需求，所以业务里最常用。

面试里最稳的回答是：
- 网关和应用入口通常优先令牌桶。
- 需要精确统计窗口行为时用滑动窗口。
- 需要平滑发送给下游时用漏桶。

### 💻 核心代码
```java
@Component
public class TokenBucketLimiter {

    private final long capacity = 200; // 桶容量
    private final long refillPerSecond = 100; // 每秒补充令牌数

    private final AtomicLong tokens = new AtomicLong(capacity);
    private final AtomicLong lastRefillTime = new AtomicLong(System.nanoTime());

    public boolean tryAcquire(int permits) {
        refillTokens(); // 先补充令牌，避免令牌长期不更新
        while (true) {
            long current = tokens.get();
            if (current < permits) {
                return false; // 令牌不足，直接限流
            }
            if (tokens.compareAndSet(current, current - permits)) {
                return true; // CAS 扣减，避免加锁
            }
        }
    }

    private void refillTokens() {
        long now = System.nanoTime();
        long last = lastRefillTime.get();
        long delta = now - last;
        long refill = delta / 1_000_000_000L * refillPerSecond; // 按秒补充
        if (refill <= 0) {
            return;
        }
        if (lastRefillTime.compareAndSet(last, now)) {
            tokens.updateAndGet(old -> Math.min(capacity, old + refill)); // 桶满则截断
        }
    }
}

@RestController
public class OrderController {

    private final TokenBucketLimiter limiter = new TokenBucketLimiter();

    @PostMapping("/orders")
    public ResponseEntity<String> createOrder() {
        if (!limiter.tryAcquire(1)) {
            return ResponseEntity.status(429).body("System busy, try later");
        }
        return ResponseEntity.ok("accepted");
    }
}
```

### ⚠️ 常见误区
- 误区 1：只在网关层限流，应用内部完全不隔离。结果是入口过了，内部还是被慢调用拖死。
- 误区 2：限流后统一返回 500。这样既不友好，也无法表达系统处于“受控降载”状态。
- 误区 3：阈值写死在代码里，不支持动态调整。高峰期根本来不及救火。

### 🎤 面试简答版
我理解的限流不是单点算法，而是分层治理。边缘层先挡异常洪水，网关层做通用限流，应用层按热点业务对象做精准限流，服务内部再通过线程池隔离避免相互拖垮。算法上我更常用令牌桶，因为它允许适度突发，比较贴近线上场景。限流一定要和降级、熔断、动态配置联动，否则只是“拦请求”，不是完整的稳定性方案。

## Q：如何设计一个高并发抢红包系统，既公平又高性能？
### 📌 一句话结论
抢红包系统的关键是把“金额分配”和“领取动作”解耦：红包发出时预生成红包分片，抢的时候只做原子领取，这样可以兼顾高并发性能和金额合法性。

### 🧠 原理讲解
抢红包最怕两件事：
- 同一份钱被多个人抢到，金额超发。
- 所有人来抢时，每次都实时计算金额，导致热点竞争严重。

本质上它和秒杀很像，都是“有限资源被高并发争抢”。区别在于秒杀是统一库存，而红包是多份金额分片。

所以最优思路不是抢的时候再算，而是发红包时就把钱拆好。这样抢的时候不做复杂计算，只做“拿走一片”的原子动作，性能会稳定得多。

### ✅ 完整解决方案
#### 1. 红包模型先定义清楚
红包一般有两类：
- 普通红包：每份金额固定。
- 拼手气红包：每份金额随机，总金额固定。

高级面试里通常问的是拼手气红包，因为它同时考察：
- 随机算法是否合理。
- 并发领取是否安全。
- 热点数据如何承载。

#### 2. 发红包时预拆分金额
红包创建时，先把总金额拆成 N 份，然后写入 Redis List 或 ZSet。

常见拆分算法：
- 二倍均值法：每次随机 `1 ~ 剩余均值 * 2`。
- 线段切分法：先生成若干随机切点，再做区间差值。

业务上通常还要加约束：
- 每份最少 0.01 元。
- 金额保留两位小数。
- 最后一份兜底收尾，保证总额精确闭合。

Trade-off：
- 二倍均值法实现简单，随机感强，但不保证严格均匀。
- 线段切分法更可控，适合对分布有更高要求的场景。

#### 3. 抢的时候只做原子弹出
红包被创建后：
- Redis 里保存红包分片列表。
- 用户抢红包时，只执行一次原子 `pop`。
- 取到金额则表示成功，取不到表示已抢完。

这样做的关键价值是：
- 抢的时候不需要实时计算金额。
- 不需要锁整张红包表。
- Redis 单线程天然保证操作串行化。

#### 4. 一人最多抢一次
抢红包的典型并发漏洞是同一用户重复点击，抢到多次。

解决方式：
- Redis Set 记录已抢用户。
- 用 Lua 脚本把“是否抢过 + 弹出金额 + 记录结果”合成一个原子操作。

红包系统很适合用 Lua，因为核心逻辑短而且要求强原子性。

#### 5. 领取结果异步落库
红包领取成功后，不建议同步写多张表：
- 红包领取记录表。
- 用户资产变更表。
- 账户流水表。
- 通知表。

更推荐：
- Redis 返回领取结果。
- 通过 MQ 异步入库和发通知。
- 钱包系统做最终入账。

如果是强资金属性场景，比如企业红包或现金提现，就不能只停留在 Redis，需要和账户系统做账务流水对账。

#### 6. 如何理解“公平”
面试里说公平，通常分三层：
- 金额公平：总金额不超发，单份金额合法。
- 机会公平：谁先抢到谁成功，不因为锁竞争造成异常偏置。
- 系统公平：不能让脚本党靠并发刷接口抢走大部分红包。

所以红包系统除了算法，还要做风控：
- 限设备、限账号、限 IP。
- 人机校验。
- 异常账号识别。

#### 7. 大群、大主播场景如何抗热点
如果一个红包会被几十万人同时抢，要进一步做热点隔离：
- 红包 key 单独分片，避免和普通缓存混部。
- 使用 Redis Cluster，热点业务单独集群。
- 结果查询走本地缓存或异步回查，避免所有用户都立刻查 DB。

再进一步，可以把大红包拆成多个子红包桶，先按路由桶分流，再在桶内抢。

#### 8. 失败补偿与状态机
抢红包成功后，如果异步落库失败怎么办？

标准做法：
- 先记领取事件日志。
- 消费失败进入重试队列。
- 超过阈值进入死信队列。
- 对账任务根据 Redis 记录和 DB 记录做核对修复。

红包如果涉及真金白银，永远不要只说“Redis 成功了就行”，面试官会直接追你账务一致性。

### 💻 核心代码
```java
@Service
public class RedPacketService {

    private final StringRedisTemplate redisTemplate;

    // KEYS[1] 红包金额列表，KEYS[2] 已领取用户集合，KEYS[3] 用户领取结果 Hash
    private static final DefaultRedisScript<String> GRAB_SCRIPT = new DefaultRedisScript<>(
            """
            local packetListKey = KEYS[1]
            local userSetKey = KEYS[2]
            local resultKey = KEYS[3]
            local userId = ARGV[1]

            if redis.call('sismember', userSetKey, userId) == 1 then
                return 'DUPLICATE' -- 同一用户只能抢一次
            end

            local amount = redis.call('lpop', packetListKey)
            if not amount then
                return 'EMPTY' -- 红包已经抢完
            end

            redis.call('sadd', userSetKey, userId) -- 标记已领取
            redis.call('hset', resultKey, userId, amount) -- 记录领取结果，便于查询与补偿
            return amount
            """,
            String.class
    );

    public String grab(Long redPacketId, Long userId) {
        String listKey = "rp:list:" + redPacketId;
        String userSetKey = "rp:users:" + redPacketId;
        String resultKey = "rp:result:" + redPacketId;

        return redisTemplate.execute(
                GRAB_SCRIPT,
                List.of(listKey, userSetKey, resultKey),
                String.valueOf(userId)
        );
    }
}
```

### ⚠️ 常见误区
- 误区 1：抢的时候再实时算随机金额。并发一高，计算和锁竞争都会成为瓶颈。
- 误区 2：只记录领取结果，不做用户去重。结果同一用户可能抢多次。
- 误区 3：把红包系统当普通缓存问题，不考虑账务补偿和对账。真金白银场景这是致命问题。

### 🎤 面试简答版
抢红包我一般会把“拆红包”和“抢红包”分开。发红包时先把总金额拆成多份写入 Redis，抢的时候只做原子弹出，同时在 Lua 里控制一人只能抢一次，这样性能和并发安全都比较好。领取成功后我不会同步做复杂落库，而是通过 MQ 异步写领取记录和资金流水。要是涉及真钱，我还会补充对账和补偿机制，因为 Redis 成功不等于账务闭环成功。

## Q：如何设计一个大流量短链系统，既能高可用跳转又能抗热点？
### 📌 一句话结论
短链系统的核心挑战不是“生成一个短码”，而是“让海量跳转请求以极低延迟完成解析，同时热点短链不能把缓存和数据库打穿”，典型方案是“短码生成防冲突 + 读路径缓存优先 + 热点隔离 + 统计异步化”。

### 🧠 原理讲解
短链看起来简单，本质上是个高读、热点极强、对 RT 很敏感的系统。

用户点击短链时，链路要非常短：
1. 根据短码查到原始长链。
2. 校验是否过期、是否禁用。
3. 立刻 302/301 跳转。

如果这一步每次都查数据库，那高峰期很容易被打穿。尤其是某个热点短链被大 V 转发后，一条链可能瞬间带来几十万 QPS。

所以短链系统真正难的是“读链路极致优化”和“统计链路解耦”。

### ✅ 完整解决方案
#### 1. 短码生成要先解决唯一性和长度问题
短码生成常见有三类方案：
- 自增 ID + Base62 编码。
- 发号器 / 雪花算法 + Base62。
- 长链摘要后截断，再做冲突处理。

真实业务里更推荐“分布式唯一 ID + Base62”，因为：
- 可控，冲突概率低。
- 生成稳定，不依赖全库查重。
- 易于水平扩展。

如果还要求“支持自定义短链”，那就要单独做唯一索引和敏感词校验。

#### 2. 跳转链路必须缓存优先
跳转是读多写少的典型场景，正常读路径应该是：
- 先查本地缓存或 Redis。
- 命中则直接返回长链。
- 未命中再查 DB，并回填缓存。

缓存内容通常包括：
- 原始长链。
- 生效状态。
- 过期时间。
- 风险控制状态。

这里要注意，跳转接口通常 RT 要求很高，所以很多系统会进一步加本地缓存，减少网络 hop。

#### 3. 热点短链要单独治理
某个明星短链爆火时，所有请求都在打一个 key，这就是典型热点 key。

解决思路一般分三层：
- 本地缓存兜底，减少 Redis 压力。
- 热点 key 永不过期或逻辑过期，避免瞬时击穿。
- 热点短链统计异步化，跳转接口不做重计算。

如果热点极强，还可以做：
- 短码分桶映射。
- 单独热点 Redis 集群。
- Nginx / CDN 层缓存跳转结果。

#### 4. 统计链路必须异步解耦
短链系统往往不只是跳转，还要统计：
- PV
- UV
- 地域
- 设备
- 来源渠道
- 时间分布

这些统计都不能同步做在跳转主链路里，否则 RT 会明显变差。

成熟做法：
- 跳转成功后只异步投递埋点消息。
- 实时统计和离线报表分层处理。
- 明细日志进 MQ/Kafka，聚合结果进 ClickHouse、Doris 或 ES。

也就是说，短链系统表面是一个跳转服务，实际上经常是“跳转服务 + 埋点分析服务”的组合。

#### 5. 防刷与安全校验不能忽略
短链天然容易被用于钓鱼、刷量、黑灰产投放，所以要有安全治理：
- 创建短链时做域名白名单或安全检测。
- 跳转前校验黑名单、封禁状态、过期状态。
- 统计侧过滤爬虫和异常流量。

如果没有这层，系统可能性能没问题，但业务上不可用。

#### 6. 数据库设计与分库思路
短链表设计至少要有：
- `short_code`
- `long_url`
- `expire_time`
- `status`
- `create_user`
- `biz_type`

高并发读主要靠缓存，DB 更多承担：
- 创建时写入。
- 缓存 miss 回源。
- 管理后台查询。

当数据量上亿时，短链映射表可以按哈希分库分表。因为查询条件主要是 `short_code`，天然适合按短码哈希路由。

#### 7. 一致性与缓存失效策略
短链被修改、禁用、延期时，缓存怎么办？

推荐做法：
- 先更新 DB。
- 再删除缓存，而不是直接更新缓存。
- 对热点 key 做延迟双删或消息通知失效。

原因是短链更新频率低、读频率高，删除缓存比维护复杂的双写一致性更稳。

#### 8. 高可用设计
短链系统是高读核心路径，故障时影响用户跳转，必须强调高可用：
- 服务无状态，多副本部署。
- Redis 高可用部署。
- 跳转核心逻辑尽量短，减少依赖。
- 非核心统计失败不影响跳转成功。

面试里这句话很加分：短链系统要优先保证“能跳”，再考虑“统计全不全”。

### 💻 核心代码
```java
@Service
public class ShortLinkResolveService {

    private final StringRedisTemplate redisTemplate;
    private final ShortLinkRepository shortLinkRepository;
    private final ApplicationEventPublisher eventPublisher;

    public String resolve(String shortCode) {
        String cacheKey = "short-link:" + shortCode;

        String longUrl = redisTemplate.opsForValue().get(cacheKey);
        if (longUrl != null) {
            eventPublisher.publishEvent(new ShortLinkVisitedEvent(shortCode)); // 异步记统计
            return longUrl;
        }

        ShortLinkDO link = shortLinkRepository.findByShortCode(shortCode)
                .filter(ShortLinkDO::isActive) // 校验是否启用
                .filter(item -> !item.isExpired()) // 校验是否过期
                .orElseThrow(() -> new NotFoundException("short link not found"));

        redisTemplate.opsForValue().set(
                cacheKey,
                link.getLongUrl(),
                Duration.ofMinutes(30) // 短链映射缓存 30 分钟
        );

        eventPublisher.publishEvent(new ShortLinkVisitedEvent(shortCode)); // 统计和跳转主链路解耦
        return link.getLongUrl();
    }
}

@RestController
public class ShortLinkController {

    private final ShortLinkResolveService resolveService;

    @GetMapping("/{shortCode}")
    public void redirect(@PathVariable String shortCode, HttpServletResponse response) throws IOException {
        String longUrl = resolveService.resolve(shortCode);
        response.setStatus(HttpServletResponse.SC_FOUND); // 302 跳转
        response.setHeader("Location", longUrl);
    }
}
```

### ⚠️ 常见误区
- 误区 1：把统计逻辑同步放在跳转主链路里。高峰时 RT 会明显变差。
- 误区 2：缓存 miss 后所有请求一起回源 DB，不做热点保护。典型缓存击穿。
- 误区 3：只考虑生成短链，不考虑黑灰产、恶意跳转和刷量风险。业务上很容易踩雷。

### 🎤 面试简答版
短链系统我会把重点放在读链路优化上，因为真正高并发的是跳转不是创建。核心做法是短码映射走缓存优先，热点短链加本地缓存和热点隔离，保证跳转接口足够短，只做解析、校验和 302 返回。访问统计我会彻底异步化，发消息做埋点聚合，避免拖慢主链路。系统设计上要优先保证“能跳”，统计不全可以补，但跳转慢或者跳不了，用户直接感知最强。

## ⚡ 快速复习索引
### 1. 如何设计一个秒杀系统，既抗住高并发又绝不超卖？
秒杀系统我一般会拆成四层：入口静态化、资格和风控前置、多级限流、核心链路异步化。库存不会直接打 MySQL，而是先在 Redis 用 Lua 做原子预扣，同时控制一人一单，成功后再发 MQ 异步创建订单。数据库只做最终账本和状态流转，失败靠补偿回滚。这个方案本质上是尽量把无效请求挡在外面，把有效请求变成有序处理，宁可少卖也不能超卖。

### 2. 如何设计一套多级限流体系，防止系统被突发流量打垮？
我理解的限流不是单点算法，而是分层治理。边缘层先挡异常洪水，网关层做通用限流，应用层按热点业务对象做精准限流，服务内部再通过线程池隔离避免相互拖垮。算法上我更常用令牌桶，因为它允许适度突发，比较贴近线上场景。限流一定要和降级、熔断、动态配置联动，否则只是“拦请求”，不是完整的稳定性方案。
